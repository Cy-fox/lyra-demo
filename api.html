API & Database Reference

Lyra exposes a REST API from the same Flask process that runs the trading engine. All state, bots, trades, logs, ML predictions, lives in a single SQLite database with WAL enabled, so the dashboard, the Telegram monitor and the reporting CLI read the same source of truth.

Overview

Base URL
http://127.0.0.1:5000, bound to localhost, reachable remotely over the VPN only
Format
JSON in, JSON out. Form-encoded also accepted on /login.
Auth
Cookie-based (auth_token, HttpOnly, SameSite=Lax, 30-day expiry)
Errors
401 unauthorized · 404 unknown bot · 429 login rate-limited

Authentication

POST/loginPassword login, sets auth cookie
Accepts {"password": "…"} as JSON or a form field. On success it sets the auth_token cookie (HttpOnly, 30-day expiry) and returns {"success": true}. Five failed attempts from the same IP lock it out for five minutes (429).
GET/logoutClears the cookie, redirects to /login
Invalidates the session by clearing the auth_token cookie and redirects the browser back to the login page. No body or parameters.

Send the password as JSON or a form field. On success the response sets the auth_token cookie; every /api/* route requires it and returns 401 otherwise. Failed attempts are rate-limited per IP, five misses lock the address out for five minutes (429).

# login and store the cookie
curl -c cookies.txt -X POST http://127.0.0.1:5000/login \
     -H "Content-Type: application/json" \
     -d '{"password": "…"}'

# authenticated request
curl -b cookies.txt http://127.0.0.1:5000/api/bots

System

GET/api/healthPer-bot liveness + system status
Returns every registered bot with its thread running flag, current status and last_update timestamp, the same feed the watchdog uses to flag stale bots. See the example response below the group.
GET/api/statusEngine status summary
Aggregate snapshot of the engine: number of bots per market, how many are running or in position, combined equity and total P&L. Cheap to poll; the dashboard header refreshes from it.
GET/api/status/<pair>Status for a single pair
Same shape as /api/status but scoped to one instrument, e.g. /api/status/BTC_USDT. Returns 404 if no bot trades that pair.
GET/api/settingsRead global settings
Returns the global key-value configuration (rows of the settings table) as one JSON object: risk defaults, watchdog thresholds, Telegram alerting toggles.
POST/api/settingsUpdate global settings
Partial update: send only the keys you want to change and they are merged into the stored settings. Values are validated before being persisted; unknown keys are rejected.
GET/api/alerts/systemStale-bot watchdog alerts
Lists bots whose last_update is older than the watchdog threshold, usually a dead thread or a stuck data feed. Empty list means all bots are healthy.
GET/api/activity/recentRecent trades and log events across bots
Merged feed of the latest closed trades and log events across every bot, newest first. Powers the dashboard activity panel; supports a limit query parameter.

/api/health reports every registered bot with its thread liveness and last update timestamp, the same feed the watchdog uses to flag stale bots:

{
  "bots": {
    "btc_usdt":  { "running": true, "status": "waiting",     "last_update": "2026-07-07T09:47:35" },
    "fut_gold":  { "running": true, "status": "in_position", "last_update": "2026-07-07T09:47:30" }
  }
}

Bots

GET/api/botsList all bots with live P&L
Array of bot summaries: market type, current price, capital vs. initial capital, open position (if any) and win/loss counters. See the example object below the group.
GET/api/bot/<bot_id>Full detail: config, position, trades, sessions
Everything about one bot: full strategy config, open position with entry/stop/TP, closed-trade history, recent analyses and (for forex) the active sessions. 404 for an unknown bot_id.
POST/api/bot/createCreate a crypto bot
Body: {"pair": "BTC/USDT", "timeframe": "1h", "mode": "conservative", "capital": 10000}. Creates the bot in created state; it only trades after /start. Returns the new bot_id.
POST/api/bot/launchOptimize-then-launch pipeline
One-shot pipeline: runs the optimizer on the requested pair, applies the best out-of-sample parameters, creates the bot and starts it. Same body as /api/bot/create plus optional optimizer mode.
POST/api/bot/<bot_id>/startStart the trading thread
Spawns the bot's trading thread; status moves from created/stopped to waiting. Idempotent: starting an already-running bot is a no-op.
POST/api/bot/<bot_id>/stopStop the trading thread
Signals the thread to stop after the current cycle. An open position is kept and resumes management on the next start; no orders are force-closed.
POST/api/bot/<bot_id>/resetReset capital and history
Restores capital to initial_capital and wipes the bot's trades, analyses and logs. The strategy config is kept. Destructive: the dashboard asks for confirmation first.
POST/api/bot/<bot_id>/sessionsSet active forex sessions
Body: {"sessions": ["london", "new_york"]}. Overrides the auto-detected trading sessions for a forex bot; entries are only taken inside these windows, exits are always allowed.
GET/api/bot/<bot_id>/projectionEquity projection cone
Projects the equity curve forward from the bot's realized trade statistics, returning median and percentile bands (the cone drawn on the bot detail chart). Needs a minimum trade history to be meaningful.
DELETE/api/bot/<bot_id>Delete bot and its history
Stops the thread if running and deletes the bot row; trades, analyses and logs cascade via foreign keys. Irreversible.

Each entry in /api/bots carries the market type (crypto, indices, forex, futures), current price, open position if any, and running win/loss counters:

{
  "id": "btc_usdt",
  "pair": "BTC/USDT",
  "market": "crypto",
  "timeframe": "1h",
  "mode": "conservative",
  "status": "waiting",
  "capital": 10000.0,
  "initial_capital": 10000.0,
  "profit_pct": 0.0,
  "has_position": false,
  "position_side": null,
  "total_trades": 0, "wins": 0, "losses": 0
}

Markets & scanning

GET/api/marketsAvailable market types
Lists the supported market types (crypto, indices, forex, futures) with the bot count and data source for each. The dashboard uses it to build the market tabs.
GET/api/pairsTradable crypto pairs
Crypto pairs available on the exchange feed (e.g. BTC/USDT, ETH/USDT). Populates the pair selector in the bot-creation form.
GET/api/indicesIndex instruments (S&P 500, NASDAQ, …)
Supported index instruments with their symbols and metadata (exchange hours, tick size). Static list; prices come from /api/indices/prices.
GET/api/indices/pricesLive index prices
Latest quote for every index instrument in one call: price, daily change and whether the underlying market is currently open.
POST/api/indices/scanScan indices for entry setups
Runs the strategy across all indices and returns the ones currently showing an entry setup, ranked by signal confidence. Body optionally narrows the timeframe or instrument list.
POST/api/indices/backtestBacktest a strategy on an index
Body: symbol, timeframe and strategy parameters. Replays historical candles and returns the metrics (return, win rate, max drawdown, trade list). Runs synchronously, expect a few seconds.
POST/api/indices/bot/createCreate an index bot
Same shape as /api/bot/create but for an index instrument; the bot only trades while the underlying market is open.
GET/api/forexForex pairs and categories
Forex pairs grouped by category (majors, minors, exotics) with pip size and typical spread per pair.
GET/api/forex/pricesLive forex quotes
Latest bid/ask for every supported forex pair, plus which trading sessions are active right now.
GET/api/forex/sessions/<pair>Auto-detected trading sessions
Returns the sessions (London, New York, Tokyo, Sydney) where the pair historically trades best, the default applied when a forex bot is created. Override per bot via /api/bot/<bot_id>/sessions.
POST/api/forex/bot/createCreate a forex bot (session-gated)
Creates a forex bot with sessions auto-detected for the pair. Entries are gated to the active sessions; exits are always allowed. Body mirrors /api/bot/create.
GET/api/futuresFutures instruments (gold, copper, …)
Supported commodity futures with contract metadata: symbol, tick value and exchange trading hours.
GET/api/futures/pricesLive futures quotes
Latest price and daily change for every futures instrument, with the market-open flag per contract.
POST/api/futures/bot/createCreate a futures bot
Creates a bot on a futures instrument (e.g. fut_gold). Same body as /api/bot/create; trading is restricted to the contract's exchange hours.

Forex bots only open positions inside their configured sessions (London, New York, Tokyo, Sydney); exits are always allowed. Sessions are auto-detected per pair at creation and can be overridden through /api/bot/<bot_id>/sessions.

Optimizer

GET/api/optimizer/modesAvailable optimization modes
Lists the optimization presets (e.g. quick / balanced / thorough) with the number of parameter combinations and the training-window length each one uses.
POST/api/optimizer/startStart an optimization run for a bot
Body: {"bot_id": "…", "mode": "balanced"}. Kicks off a background run for an existing bot; parameters are fit on a training window and scored out-of-sample. Poll progress via /api/optimizer/status/<bot_id>.
GET/api/optimizer/status/<bot_id>Poll run progress
Progress of the bot's current run: percent complete, best score so far and, once finished, the candidate parameter set awaiting /api/optimizer/apply.
POST/api/optimizer/applyApply the optimized parameters
Body: {"bot_id": "…"}. Writes the finished run's best parameters into the bot's config (stored in optimization_json). Nothing changes until you apply; runs never auto-apply.
POST/api/bot/optimizeStandalone optimization job
Optimizes a pair/timeframe without needing an existing bot, used by the launch pipeline and for research. Returns a job_id to fetch results or stream progress.
GET/api/bot/optimize/<job_id>Job result
Final result of a standalone job: best parameters, in-sample vs. out-of-sample scores and the backtest metrics behind them. 404 if the job id is unknown or expired.
GET/api/bot/optimize/<job_id>/streamProgress as server-sent events
Server-sent events stream (text/event-stream) emitting one event per tested combination, so the dashboard shows a live progress bar without polling. Closes when the job finishes.

Optimization runs out-of-sample: parameters are fit on a training window and scored on unseen candles. Long-running jobs stream progress over SSE so the dashboard can show a live progress bar without polling.

Risk & ML

GET/api/risk/portfolioPortfolio-level exposure and correlation
Portfolio view across all bots: total exposure per market, open-position correlation and combined drawdown. This is the check that stops several bots from stacking the same directional bet.
GET/api/risk-profilesRisk profile presets
The conservative / moderate / aggressive presets with their concrete values: risk per trade, stop distance and maximum concurrent positions.
GET/api/portfolio/dailyDaily portfolio equity series
Time series of combined equity across all bots, one point per day. This is the data behind the portfolio chart and the weekly report.
GET/api/ml/statusMeta-labeling gate status per market
Per-market state of the ML gate: whether a trained LightGBM model is loaded, its threshold, training-set size and counts of taken vs. skipped signals.
GET/api/ml/calibrationP(win) calibration buckets
Buckets resolved predictions by predicted P(win) and compares each bucket with the realized hit rate; a well-calibrated model tracks the diagonal. Built from the ml_predictions table after the daily resolver runs.
POST/api/ml/create_test_botsSpin up A/B bots (ML gate on vs. off)
Creates a paired experiment: two identical bots on the same pair, one with the ML gate enabled and one without, so the gate's edge can be measured on live signals instead of backtests.

The ML layer is a meta-labeling gate: pooled per-market LightGBM models score each strategy signal with a win probability, and signals below the threshold are skipped. Every prediction is persisted (see ml_predictions below) so calibration - predicted P(win) vs. realized hit rate, can be audited after trades resolve.

Database, engine

Engine
SQLite 3, single file (lyra.db)
Journal
WAL, dashboard reads never block the trading threads' writes
Durability
synchronous=NORMAL, busy_timeout=5000
Integrity
foreign_keys=ON; child rows cascade on bot deletion
Migration
Imported from the legacy per-bot JSON state files via migrate_json_to_sqlite.py
